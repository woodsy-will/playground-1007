## Agent Team Architecture

The development workflow is driven by a five-agent team. Each agent owns a domain of responsibility and communicates through structured handoff protocols. The orchestrator routes tasks, manages dependencies, and enforces quality gates.

| Agent | Role | Owns |
|---|---|---|
| **Orchestrator** | Task decomposition, dependency tracking, cross-project scheduling | `agents/`, CI/CD, milestone tracking |
| **Data Engineer** | Acquisition, preprocessing, schema validation, pipeline plumbing | `shared/`, data download scripts, ETL |
| **Spatial Analyst** | Core geospatial logic — raster math, vector ops, LiDAR processing | `projects/*/src/`, algorithm implementation |
| **ML Engineer** | Model training, validation, hyperparameter tuning, experiment tracking | Model code, evaluation scripts, metrics |
| **Technical Writer** | Documentation, white papers, README maintenance, visualization polish | `projects/*/docs/`, notebooks, figures |

### Handoff Protocol

1. **Orchestrator** decomposes a user request into atomic tasks and assigns to agents.
2. Each agent produces output with a **manifest** — a structured log of inputs consumed, outputs produced, assumptions made, and known limitations.
3. Downstream agents consume upstream manifests before starting work. If an upstream output is missing or invalid, the downstream agent raises a **block** back to the Orchestrator rather than proceeding with assumptions.
4. The **Technical Writer** produces deliverables only after Spatial Analyst and ML Engineer outputs pass validation.

### Quality Gates

Each phase transition requires:

- [ ] All tests pass (`pytest` for Python, `testthat` for R)
- [ ] No hardcoded paths — all paths resolved from config
- [ ] CRS consistency verified (default: EPSG:3310, California Albers NAD 83)
- [ ] Outputs reproducible from config + data + code alone

## Project Guides

### P3 — Individual Tree Crown Delineation & Biometric Extraction

**Goal:** Process raw LiDAR point clouds to delineate individual tree crowns and extract per-tree structural biometrics, then validate against field cruise data.

**Phases:**

| Phase | Task | Agent | Key Outputs |
|---|---|---|---|
| 1 | Data acquisition — download 3DEP QL1 tiles, select validation plots | Data Engineer | LAZ tiles, plot coordinates, cruise data |
| 2 | Ground classification (SMRF via PDAL) → DTM (1.0 m) → CHM | Spatial Analyst | DTM raster, CHM raster |
| 3 | CHM smoothing → local maxima detection (variable window) → watershed segmentation | Spatial Analyst | Treetop points, crown polygons |
| 4 | Per-tree metric extraction: height, crown area, crown diameter | Spatial Analyst | Tree attribute table |
| 5 | Allometric modeling — DBH from height + crown width, stand-level rollup (TPA, BA/ac, QMD, SDI) | ML Engineer | Biometric predictions with uncertainty |
| 6 | Validation — spatial join to cruise stems, omission/commission rates, RMSE by species/diameter class | ML Engineer | Accuracy assessment report |
| 7 | 3D interactive visualization, technical white paper | Technical Writer | HTML viz, PDF white paper |

**Key Parameters (configs/default.yaml):**
```yaml
dtm_resolution: 1.0        # meters
chm_smoothing_sigma: 0.67  # Gaussian kernel sigma (meters)
min_tree_height: 5.0        # meters — suppress understory false positives
allometric_source: "pillsbury_kirkley_1984"
crs: "EPSG:3310"
```

**Critical Risk:** High commission errors in dense multi-layered canopy. Mitigate with height thresholds and minimum crown area filters.

### P1 — Automated Post-Wildfire Burn Severity & Recovery Tracker

**Goal:** Automated pipeline that ingests pre/post-fire satellite imagery, classifies burn severity via dNBR, and tracks vegetation recovery over consecutive years.

**Phases:**

| Phase | Task | Agent | Key Outputs |
|---|---|---|---|
| 1 | Fire selection — 2–3 recent Sierra Nevada fires with known perimeters (MTBS/CAL FIRE FRAP) | Data Engineer | Fire perimeter vectors, metadata |
| 2 | Image acquisition — Sentinel-2 L2A via Copernicus Data Space API, cloud filter < 20% | Data Engineer | Pre-fire, post-fire, and annual anniversary scenes |
| 3 | NBR computation, dNBR derivation, severity classification (USGS/BAER thresholds) | Spatial Analyst | Severity raster (4-class) |
| 4 | NDVI/EVI time series extraction per severity class, recovery curve fitting | ML Engineer | Recovery trajectories, time-to-recovery estimates |
| 5 | Zonal statistics — area by severity within management units, estimated volume impact | Spatial Analyst | Summary tables per unit |
| 6 | Interactive dashboard (Plotly Dash), technical white paper | Technical Writer | Dashboard app, PDF white paper |

**Key Parameters (configs/default.yaml):**
```yaml
severity_thresholds:
  unburned: [-0.1, 0.1]
  low: [0.1, 0.27]
  moderate_low: [0.27, 0.44]
  moderate_high: [0.44, 0.66]
  high: [0.66, 1.3]
cloud_cover_max: 20          # percent
recovery_index: "NDVI"       # or "EVI"
crs: "EPSG:3310"
```

**Critical Risk:** Cloud/snow contamination in Sierra Nevada winter scenes. Mitigate with seasonal compositing and Landsat backup.

### P4 — Dynamic Habitat Suitability Modeling under Climate Scenarios

**Goal:** Model current and future habitat suitability for a sensitive focal species using topographic, climatic, and vegetative predictors. Project onto CMIP6 climate scenarios to identify refugia and range shifts.

**Phases:**

| Phase | Task | Agent | Key Outputs |
|---|---|---|---|
| 1 | Species selection, occurrence data acquisition (GBIF, CNDDB), spatial thinning | Data Engineer | Cleaned occurrence points |
| 2 | Predictor compilation — DEM derivatives, bioclimatic variables (WorldClim/PRISM), canopy cover | Data Engineer | Aligned predictor raster stack |
| 3 | Model fitting — MaxEnt + Random Forest, blocked spatial cross-validation | ML Engineer | Fitted models, AUC/TSS metrics |
| 4 | Variable importance, partial dependence plots | ML Engineer | Feature importance charts |
| 5 | Future projection — CMIP6 SSP2-4.5 and SSP5-8.5 at mid-century and end-of-century | Spatial Analyst | Projected suitability rasters |
| 6 | Refugia mapping, suitability change analysis (gain/loss/stable) | Spatial Analyst | Change detection maps |
| 7 | Publication-quality cartographic output, technical white paper | Technical Writer | Map series, PDF white paper |

**Key Parameters (configs/default.yaml):**
```yaml
focal_species: "Pekania pennanti"    # Pacific fisher — adjust as needed
thinning_distance_km: 1.0
cv_method: "spatial_block"
cv_folds: 5
scenarios: ["ssp245", "ssp585"]
time_steps: [2050, 2090]
crs: "EPSG:3310"
```

**Critical Risk:** MaxEnt overfitting to sampling bias. Mitigate with target-group background sampling and blocked CV.

### P2 — Local LLM-Powered Spatial Query Interface

**Goal:** Natural language interface that translates user queries into spatial SQL/ArcPy operations against a file geodatabase, returning map layers or summary tables.

**Phases:**

| Phase | Task | Agent | Key Outputs |
|---|---|---|---|
| 1 | Schema design — target FGDB with harvest units, streams, roads, sensitive habitats, ownership | Data Engineer | Populated geodatabase |
| 2 | LLM deployment — quantized open-source model (Llama 3 8B or Mistral 7B) on Proxmox | ML Engineer | Running inference endpoint |
| 3 | RAG corpus — schema metadata, few-shot query examples, spatial predicate definitions | ML Engineer | Retrieval index |
| 4 | Query translation layer — text-to-SQL with validation (whitelist SELECT, reject destructive ops) | Spatial Analyst | Translation module with guardrails |
| 5 | Execution engine — run validated queries against FGDB, return layers or tables | Spatial Analyst | Query executor |
| 6 | Feedback loop — zero-result reformulation, error handling | ML Engineer | Retry logic |
| 7 | Accuracy assessment (50+ test queries), architecture diagram, screencast demo | Technical Writer | Evaluation report, demo video |

**Key Parameters (configs/default.yaml):**
```yaml
llm_model: "meta-llama/Meta-Llama-3-8B-Instruct"
quantization: "GPTQ-4bit"
max_tokens: 512
allowed_operations: ["SELECT", "ST_Buffer", "ST_Intersects", "ST_Within", "ST_Contains"]
blocked_operations: ["DELETE", "DROP", "UPDATE", "INSERT", "ALTER"]
geodatabase_path: "data/project.gdb"
```

**Critical Risk:** LLM hallucinating nonexistent field names. Mitigate with strict schema injection in every prompt and constrained decoding.
